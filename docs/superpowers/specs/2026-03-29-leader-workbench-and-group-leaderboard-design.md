# 组长督战工作台与小组榜单设计

## 1. 背景

当前项目已经完成了校园卡新流程的 phase-1.5 基础闭环：

- `ADMIN / LEADER / MEMBER` 三角色已经接通
- 管理员可导入识别码和新生 `QQ号` 线索，并按成员分发
- 成员可围绕 `IdentifierCode + IdentifierSale` 完成成交
- 旧 `SalesRecord` 仍作为排行榜、结算、导出的兼容汇总来源

但组长侧和小组维度的关键页面仍未落地：

- `/leader/sales` 仍是占位页
- `/leaderboard/groups` 仍是占位页

这导致当前系统有两处明显断层：

1. 组长虽然已经有角色和路由，但没有真正可用的日常督战与调度工作台
2. 小组维度虽然已经进入导航，但没有对外可见的小组榜单和对内可追责的小组明细

本设计要补齐的不是“再做一个展示页”，而是把组长从“有角色没工具”升级成“有督战、有调度、有推进、有审计”的真实工作台，同时把共享小组榜单接到可用状态。

## 2. 本期目标

本期只聚焦一个子项目：补齐“组长督战工作台 + 小组榜单”。

交付目标如下：

1. 把 `/leader/sales` 从占位页升级为组长日常主入口
2. 把 `/leaderboard/groups` 从占位页升级为可公开访问的小组榜单
3. 让组长能在自己小组内推进线索、调度识别码、查看掉队成员和排名差距
4. 在不打断现有 `IdentifierSale -> SalesRecord` 兼容链路的前提下，建立小组维度的督战口径
5. 为所有组内调度和状态推进建立强审计链路

## 3. 已确认的产品决策

以下决策已经锁定，后续实现不得偏离：

- 本子项目优先级高于结算重写与 Vercel 真库验收
- `/leader/sales` 不是单纯销售记录页，而是“组长督战工作台”
- 首页核心场景是“当日冲榜督战”
- 首页首屏采用双列并重：
  - 左侧先看组内成员冲榜
  - 右侧同时看各组排名变化
- 榜单口径采用混合模型：
  - 今日督战优先以 `IdentifierSale` 为强事实源
  - 同时保留与旧 `SalesRecord` 对齐的兼容指标
- 第一版允许组长直接做重操作，而不只是看板和跳转
- 资源初次分发仍归管理员负责
- 组长只允许在自己小组内部进行二次调度
- 组长可以在本组内改派：
  - 线索
  - 识别码
- 组长不能：
  - 跨组调度
  - 从公共池直接拉资源
  - 代替管理员做全局分发
- 推进流采用双轨并行：
  - 正式 `QQ号` 线索推进
  - 早期无 `QQ号` 的自主获客跟进项推进
- 早期无 `QQ号` 的跟进项，第一版只记录：
  - 状态
  - 备注
- 所有改派、回收、转交、状态推进都必须进入不可篡改的强审计
- 最终成交仍只能由成员通过现有录单链路完成
- 组长不能直接手工把资源关成“已成交”
- `/leaderboard/groups` 第一版对所有登录角色开放小组级汇总
- `/leaderboard/groups` 的成员明细展开权限为：
  - `ADMIN` 可展开任意组
  - `LEADER` 只可展开自己负责的小组
  - `MEMBER` 不可展开任意组成员细节

## 4. 本期范围与明确不做

### 4.1 本期必须交付

- `/leader/sales` 组长督战工作台
- `/leaderboard/groups` 共享小组榜单
- 组内成员冲榜视图
- 小组对小组排名视图
- 正式线索推进队列
- 早期自主获客跟进队列
- 组内识别码调度队列
- 组长在本组内的线索改派和识别码改派
- 组长重操作的原因填写与确认交互
- 组内调度与推进的强审计时间线

### 4.2 本期明确不做

- 结算重写为统一基于 `IdentifierSale`
- `40 / 60` 佣金规则迁移
- 跨组调度
- 组长从公共池直接领资源
- 组长手工关单
- 售后回退、误录撤销、已转化资源重新打开
- 公共大而全趋势分析中心
- Vercel Preview 和真实 Neon 数据库联调验收

## 5. 页面与信息架构

### 5.1 `/leader/sales`

`/leader/sales` 升级为“组长督战工作台”，是 `LEADER` 的默认高频工作入口。

页面结构固定如下：

#### A. 顶部摘要条

展示组长最关心的今日摘要：

- 本组今日成交总数
- 本组 `40 / 60` 结构
- 当前小组排名
- 与上一组 / 下一组的差距
- 本组待跟进项数量
- 本组空置或待调度识别码数量

#### B. 首屏双列

左列为“组内成员冲榜”：

- 今日成交数
- `40 / 60` 拆分
- 当前持有码数
- 待推进项数
- 最近动作时间
- 掉队状态

右列为“各组排名变化”：

- 本组名次
- 所有小组当日排名
- 与相邻小组差距
- 今日新增
- 趋势方向

这两列都必须可直接引出行动：

- 从成员卡片进入成员的跟进项抽屉和识别码调度抽屉
- 从榜单卡片进入对应小组的汇总视图，成员级明细仍按角色权限裁剪

#### C. 线索推进区

这一块同时承接两类对象，但 UI 上表现为同一条工作队列：

- 正式 `QQ号` 线索
- 早期无 `QQ号` 的自主获客跟进项

默认按“最需要组长处理”排序：

- 长时间未更新
- 状态停滞
- 临近成交
- 当前无人跟进

#### D. 识别码调度区

展示组内所有可调度识别码：

- 当前持有人
- 最近分配时间
- 是否长期未动
- 是否可组内转交
- 是否已进入待成交阶段

#### E. 审计时间线

展示本组最近发生的关键动作：

- 谁把什么资源从谁转给谁
- 谁推进了什么状态
- 谁回收了什么资源
- 原因是什么

这个时间线是追责工具，不是装饰日志。

### 5.2 `/leader/group`

`/leader/group` 继续保留，但职责收缩为“本组基础信息页”。

它负责：

- 组名、口号、备注
- 成员清单
- 基础汇总
- 跳转到 `/leader/sales`

它不再承担主调度职责。

### 5.3 `/leaderboard/groups`

`/leaderboard/groups` 是共享榜单页。

公开层只展示小组级汇总：

- 排名
- 今日总成交
- `40 / 60` 拆分
- 与前后组差距
- 趋势方向

成员明细展开按角色裁剪：

- `ADMIN` 可展开任意组明细
- `LEADER` 只可展开自己所在组明细
- `MEMBER` 只能看小组级汇总，不能看任意组成员层细节

## 6. 数据模型设计

### 6.1 榜单事实源

第一版不推翻现有双轨。

榜单与督战采用混合口径：

- 主展示口径：`IdentifierSale`
- 兼容口径：`SalesRecord`

约束如下：

- 今日冲榜、成员表现、小组排名的主排序以 `IdentifierSale` 聚合结果为准
- 页面中仍展示 `40 / 60` 结构和与旧统计对齐的兼容指标
- 旧结算、旧导出和未重写页面继续依赖 `SalesRecord`

这样可以避免：

- 组长工作台看见的是新事实
- 旧报表和结算立即失效
- 过渡期出现两套完全断裂的口径

### 6.2 `IdentifierCode` 增补组级归属

如果组长要支持“组内回收”和“组内再分配”，仅靠当前 `currentOwnerUserId` 不足以表达“回收到本组待分配池”。

因此需要在 `IdentifierCode` 上补充组级归属字段，建议为：

- `assignedGroupId`

语义如下：

- 管理员首次把识别码分给某成员时，同时固化该识别码当前属于哪个组
- 组长在本组内转交识别码时：
  - `assignedGroupId` 不变
  - `currentOwnerUserId` 改变
- 组长把识别码回收到本组待分配池时：
  - `assignedGroupId` 保持本组
  - `currentOwnerUserId = null`
  - `status = ASSIGNED`

第一版不引入跨组识别码迁移。

### 6.3 新增 `GroupFollowUpItem`

为避免把“正式 `QQ号` 线索”和“早期无 `QQ号` 跟进对象”混成一张脏表，新增组长工作台专用过程表 `GroupFollowUpItem`。

建议字段：

- `id`
- `groupId`
- `currentOwnerUserId`
  - 可为空；为空表示当前在本组待分配池中，无具体负责人
- `sourceType`
  - `PROSPECT_LEAD`
  - `MANUAL_DISCOVERY`
- `prospectLeadId`
  - 仅在正式 `QQ号` 线索场景下有值
- `status`
  - `UNTOUCHED`
  - `FOLLOWING_UP`
  - `APPOINTED`
  - `READY_TO_CONVERT`
  - `INVALID`
  - `CONVERTED`
- `summaryNote`
- `createdByUserId`
- `lastActionAt`
- `convertedAt`
- `createdAt`
- `updatedAt`

规则：

- 正式 `QQ号` 线索可以映射为 `GroupFollowUpItem`
- 早期自主获客对象只创建 `MANUAL_DISCOVERY` 跟进项，不强制要求 `QQ号`
- 第一版早期跟进项只允许维护：
  - 状态
  - 备注
- 一个正式 `ProspectLead` 在同一时刻只能对应一个活跃跟进项

### 6.4 保留 `ProspectLead` 的正式事实地位

`ProspectLead` 仍然是正式 `QQ号` 线索表，不被 `GroupFollowUpItem` 取代。

职责划分固定如下：

- `ProspectLead`
  - 管正式 `QQ号`、专业、来源、是否已分配、是否已转化
- `GroupFollowUpItem`
  - 管组长督战和成员推进过程
- `IdentifierSale`
  - 管最终成交事实

这三张表不能混用，否则后续很难同时满足：

- 早期无 `QQ号` 跟进
- 后期正式 `QQ号` 线索
- 成交强事实

### 6.5 新增 `GroupResourceAuditLog`

所有组内重操作都必须写入强审计表 `GroupResourceAuditLog`。

建议字段：

- `id`
- `groupId`
- `operatorUserId`
- `resourceType`
  - `FOLLOW_UP_ITEM`
  - `PROSPECT_LEAD`
  - `IDENTIFIER_CODE`
- `resourceId`
- `actionType`
  - `REASSIGN`
  - `RETURN_TO_GROUP_POOL`
  - `STATUS_CHANGE`
  - `NOTE_UPDATE`
  - `CONVERTED_LINKED`
- `beforeSnapshot`
- `afterSnapshot`
- `reason`
- `createdAt`

约束如下：

- 所有会改变归属、状态、备注语义的重要动作都要写入
- 审计记录只允许追加，不允许修改和删除
- 任何没有原因的重操作都不能提交

## 7. 状态流与交互规则

### 7.1 线索推进流

线索推进统一使用工作台状态，而不扩展 `ProspectLeadStatus` 去承接全部经营语义。

`ProspectLeadStatus` 继续只表达正式线索生命周期：

- `UNASSIGNED`
- `ASSIGNED`
- `CONVERTED`

经营推进语义放在 `GroupFollowUpItem.status`：

- `UNTOUCHED`
- `FOLLOWING_UP`
- `APPOINTED`
- `READY_TO_CONVERT`
- `INVALID`
- `CONVERTED`

这样可以避免把“是否已分配”和“跟进到哪一步了”强行塞进一个枚举里。

### 7.2 早期无 `QQ号` 跟进流

当成员在前期自主获客、但暂时还没有 `QQ号` 时，组长可在 `/leader/sales` 中登记 `MANUAL_DISCOVERY` 跟进项，并把它直接分给某个成员，或先放入本组待跟进池。

第一版只要求：

- 选择当前负责人
- 选择状态
- 填写备注

不要求：

- `QQ号`
- 手机号片段
- 微信名
- 自定义字段集合

这保证工作台能真实反映前期业务，而不会逼迫成员在还拿不到 `QQ号` 时伪造数据。

第一版不额外要求先为成员端新增一套“手动新建跟进项”页面。

### 7.3 与现有成交链路的衔接

本期不改写“只有成员才能完成最终成交”这条边界。

因此成交流程保持：

1. 成员在现有 `/entry` 链路里选择识别码
2. 成员在成交时必须关联已有正式 `ProspectLead`，或把早期跟进项补齐为正式 `ProspectLead`
3. 系统写入 `IdentifierSale`
4. 系统把识别码改为 `SOLD`
5. 系统把对应 `ProspectLead` 改为 `CONVERTED`
6. 系统同步更新旧 `SalesRecord`
7. 系统把对应 `GroupFollowUpItem` 标记为 `CONVERTED`

这里的关键边界是：

- 组长工作台可以提前管理“谁在跟进”
- 但最终成交仍然在成员录单时完成
- 组长不能跳过成员录单直接关单

### 7.4 识别码调度流

识别码调度只允许发生在组内。

第一版支持三种动作：

- 成员 A 转交给成员 B
- 从成员手中回收到本组待分配池
- 从本组待分配池再分给本组成员

每次动作都必须：

- 展示变更前后对象
- 填写原因
- 写入强审计

以下场景必须禁止：

- 已售出识别码再调度
- 跨组调度
- 未绑定本组的识别码被本组组长操作

### 7.5 公开榜单与私有明细的权限裁剪

`/leaderboard/groups` 必须同时满足“公共竞争感”和“成员细节不完全公开”。

具体规则：

- 小组级汇总对所有登录角色开放
- 成员级明细只对有责任边界的人开放
- `LEADER` 只能展开自己所在组
- `ADMIN` 可展开任意组
- `MEMBER` 不显示任意展开入口

## 8. 错误处理与并发策略

第一版统一采用保守策略：拒绝覆盖，不做静默吞写。

以下情况提交必须失败，并返回明确错误：

- 资源已经被其他人改派
- 识别码已售出
- 正式线索已转化
- 当前组长试图操作非本组资源
- 跟进项已处于 `INVALID` 或 `CONVERTED` 终态仍试图继续推进
- 未填写改派或回收原因

交互要求：

- 重操作全部走显式确认层
- 确认层必须展示变更前后
- 失败时提示“状态已变化，请刷新后重试”

服务端要求：

- 所有改派与状态推进走事务
- 事务内同时完成：
  - 权限校验
  - 当前状态校验
  - 主数据更新
  - 审计追加

## 9. 测试策略

### 9.1 服务层测试

至少覆盖：

- 组长只能操作本组资源
- 组内识别码可转交、可回收、可再分配
- 正式线索与早期跟进项都可推进状态
- 非法状态跳转被拒绝
- 并发冲突时拒绝覆盖
- 每个关键动作都写入审计

### 9.2 页面与权限测试

至少覆盖：

- `LEADER` 可进入 `/leader/sales`
- `MEMBER` 无法进入 `/leader/sales`
- `/leaderboard/groups` 对三角色都可见小组级汇总
- `LEADER` 只能展开自己组明细
- `ADMIN` 可展开任意组明细
- `MEMBER` 看不到成员级展开入口

### 9.3 成交衔接测试

至少覆盖：

- 成员最终成交后，`IdentifierSale` 正常写入
- 识别码状态更新为 `SOLD`
- 正式 `ProspectLead` 更新为 `CONVERTED`
- 对应 `GroupFollowUpItem` 更新为 `CONVERTED`
- 旧 `SalesRecord` 兼容汇总继续同步

### 9.4 E2E 冒烟

至少覆盖一条组长真实路径：

1. 组长进入 `/leader/sales`
2. 查看本组成员掉队情况
3. 把一个跟进项组内转交
4. 把一个识别码组内转交或回收
5. 查看审计时间线已出现记录
6. 成员完成成交
7. 组长刷新后看到小组榜单和成员状态变化

## 10. 实现边界总结

这一版的核心不是把整个校园卡系统一次做完，而是补齐两个已经在导航里存在、却仍然空着的关键角色页面。

实现完成后，系统应达到的状态是：

- 管理员继续负责资源初次分发
- 组长真正拥有组内督战与二次调度工具
- 成员继续负责最终成交录入
- 共享榜单能公开展示小组竞争态势
- 所有关键组内动作都有可追责审计链

结算重写、跨组经营分析、真实部署验收仍然是后续子项目，不并入本次设计。
