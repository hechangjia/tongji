# 成员识别码工作台设计

## 1. 背景

管理员侧第一阶段已经完成：

- `/admin/codes` 可导入识别码
- `/admin/codes` 可导入新生 QQ 线索
- 管理员可把识别码和线索分配给成员

当前成员侧仍然停留在旧模式：

- `/entry` 只支持按天录入 `40 / 60` 数量
- `/records` 只能看按天汇总
- 识别码、线索、成交事实还没有进入成员日常链路

而实际业务已经明确需要：

- 成员围绕“自己名下识别码”完成成交
- 成交时必须保留 `40 / 60` 套餐类型
- 新生信息来源既可能来自管理员后发的 QQ 线索，也可能来自成员自己临场拿到的新生信息

## 2. 本期目标

本期只做成员侧第一阶段，不重写结算与全平台统计。

交付目标：

1. 把 `/entry` 升级成“成员工作台”
2. 新增“识别码成交”主流程
3. 保留旧的按日 `40 / 60` 数量录入，作为过渡 fallback
4. 让新成交能够桥接到现有 `SalesRecord`，保证旧榜单和结算暂时不断
5. `/records` 能看到识别码成交来源信息

## 3. 已确认的业务决策

### 3.1 `40 / 60` 不移除

`40 / 60` 继续保留，且成为每次识别码成交时必须选择的套餐类型，后续仍作为结算卡酬依据。

### 3.2 成交前不要求先手动绑定线索

成员成交时一次完成：

- 选识别码
- 选 `40 / 60`
- 选已分配线索或手填 `QQ号 + 专业`

不强制先做“识别码和线索预绑定”。

### 3.3 前期允许成员直接手填新生信息

因为新生 QQ 线索往往是后续才批量拿到，前期成员仍然会零散拿到办卡信息。

因此必须允许成员在没有线索池支持时直接录单，至少手填：

- `QQ号`
- `专业`

### 3.4 手填新生信息要自动补进线索池

如果成员成交时手填 `QQ号 + 专业`：

- 若该 QQ 不存在：自动创建一条 `ProspectLead`
- 若该 QQ 已存在：直接复用已有线索

线索池以 `QQ号` 全局去重。

## 4. 页面设计

### 4.1 `/entry`

`/entry` 升级为“成员工作台”，顶部展示：

- 我名下待售识别码
- 分配给我的新生线索
- 今日已成交单数
- 今日 `40 / 60` 汇总

主体分两块：

#### A. 识别码成交主入口

表单字段：

- `识别码`
  - 只能选择当前成员名下、且状态不是 `SOLD` 的识别码
- `套餐类型`
  - 必选 `40 / 60`
- `新生信息来源`
  - 选择已分配线索
  - 或手动填写 `QQ号 + 专业`
- `备注`
  - 可选

#### B. 旧录入 fallback

保留现有按日 `40 / 60` 数量录入入口，作为过渡方案。

但它不再是推荐入口，应在 UI 上明确标识为“旧录入模式 / 过渡使用”。

### 4.2 `/records`

继续保留“我的记录”，但在原有基础上增加：

- 识别码编号
- 套餐类型
- 新生信息来源
  - 管理员分配线索
  - 成员手填

## 5. 数据模型

### 5.1 扩展 `ProspectLeadStatus`

从当前：

- `UNASSIGNED`
- `ASSIGNED`

扩展为至少：

- `UNASSIGNED`
- `ASSIGNED`
- `CONVERTED`

用于表示线索已被某笔成交转化。

### 5.2 新增 `IdentifierSale`

新增“识别码成交事实表”，只存单笔成交事实，不混进旧的日汇总。

建议字段：

- `id`
- `codeId`
- `sellerUserId`
- `groupId`
- `prospectLeadId`
- `planType`: `PLAN_40 / PLAN_60`
- `saleDate`
- `remark`
- `createdAt`
- `updatedAt`

### 5.3 旧 `SalesRecord` 的定位

`SalesRecord` 本期不删除，但降级为“旧看板/旧结算的桥接汇总表”。

它不再是唯一事实源。

## 6. 成交数据流

成员提交识别码成交时：

1. 校验识别码属于当前成员
2. 校验识别码尚未售出
3. 解析新生信息来源
   - 若选已有线索：校验线索已分配给当前成员，且未转化
   - 若手填：按 QQ 查询现有线索，不存在则创建
4. 写入 `IdentifierSale`
5. 更新 `IdentifierCode.status = SOLD`
6. 更新对应 `ProspectLead.status = CONVERTED`
7. 反算并同步当天旧 `SalesRecord`
   - `PLAN_40` 累加到 `count40`
   - `PLAN_60` 累加到 `count60`

## 7. 过渡策略

### 7.1 双轨并存

这期采用双轨：

- 新：识别码成交事实流
- 旧：按日 `40 / 60` 汇总流

### 7.2 桥接规则

新成交保存后，会同步覆盖当天 `SalesRecord` 汇总，保证旧榜单/旧结算还能工作。

### 7.3 冲突处理

为避免同一天双轨同时写入打架，采用硬规则：

- 只要某天已经存在 `IdentifierSale`
- 就不允许再通过旧的按日录入方式修改该天汇总

这样可以避免：

- 新成交累加一套
- 旧表单再手工改另一套
- 导致当天汇总失真

## 8. 关键边界

- 成员只能成交自己名下识别码
- 已售识别码不能再次成交
- 已转化线索不能再次成交
- 手填 QQ 如已存在，必须复用已有线索
- 成交时固化 `groupId`，后续换组不影响历史归属
- 若线索来自管理员分配，成交后应能让管理员追溯“这条线索被谁转化”

## 9. 本期不做

- 组长侧识别码/线索看板
- 成员侧预发放状态流转页面
- 管理员销售修改/撤销与 `IdentifierSale` 的完全联动回滚
- 结算改造为完全基于 `IdentifierSale`

这些属于后续 phase。

## 10. 验证重点

本期实现至少需要覆盖：

- 成员只能选择自己名下识别码
- 手填 QQ 自动补进线索池且按 QQ 去重
- 已有线索可直接成交
- 新成交会同步更新旧 `SalesRecord`
- 某天一旦存在 `IdentifierSale`，旧录入入口禁止再写该天
- `/records` 能显示识别码编号与套餐类型
