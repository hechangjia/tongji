# 校园电话卡销售统计与卡酬结算系统设计

- 日期：2026-03-25
- 形态：响应式网页
- 目标用户：暑期线下校园电话卡销售团队
- 目标：支持成员每日录入销售数据、管理员查看排行与结算，并支持后续 11 月继续使用的数据持久化场景

## 1. 项目目标

构建一个轻量、可长期保存数据的销售统计与卡酬结算系统，解决以下核心问题：

1. 提供每日排行榜，统计每位成员当日卖卡数量，用于每日激励。
2. 提供总榜，支持自定义时间段统计所有成员卖卡数量。
3. 支持两种套餐（40 / 60）的独立数量统计与卡酬计算。
4. 支持管理员为每个成员分别配置卡酬单价，并允许部分时间段调整。
5. 支持导出 Excel，便于结算、复核和留档。
6. 支持暑期结束后继续保留数据，在后续时间点再次使用。

## 2. 设计原则

### 2.1 KISS
- 第一版只解决录入、统计、排行、结算、导出。
- 不做复杂审批流。
- 不做复杂财务系统。
- 不做多层级权限体系。

### 2.2 YAGNI
- 不做短信登录、微信登录、消息通知、多团队隔离、动态图表、自动发薪等非核心能力。
- 套餐种类第一版固定为 40 和 60，不做动态套餐配置。

### 2.3 DRY
- 排行榜和结算结果基于销售记录与规则实时计算，不重复维护第二份业务数据。
- 尽量通过统一数据模型支撑多个功能模块。

### 2.4 稳定优先
- 数据必须数据库持久化，不依赖本地缓存、临时文件或纯前端存储。
- 后端承担最终校验和权限控制责任。

## 3. 产品形态与总体方案

### 3.1 产品形态
采用响应式网页。

原因：
- 用户已有 Web 开发经验，交付速度最快。
- 成员主要在手机端录入，管理员主要在电脑端管理，响应式网页同时覆盖两类场景。
- 相比 Flutter，更适合表单、表格、后台管理、筛选统计类系统。

### 3.2 总体部署方案
- 前后端统一使用 TypeScript。
- 使用 Next.js 构建单仓全栈应用。
- 使用 PostgreSQL 存储业务数据。
- 使用 Prisma 作为数据库访问层。
- 部署平台使用 Vercel。
- 域名和 DNS 继续由 Cloudflare 管理。
- 使用 ExcelJS 生成 `.xlsx` 导出文件。

## 4. 核心角色与权限

第一版仅保留两种角色：

### 4.1 member
可执行：
- 登录系统。
- 新增自己的销售记录。
- 查看并编辑自己的可编辑记录。
- 查看日榜和总榜。
- 查看自己的历史记录。

不可执行：
- 修改他人记录。
- 修改卡酬规则。
- 访问管理后台。
- 导出管理员级数据。

### 4.2 admin
可执行：
- 管理成员账号。
- 查看全部销售记录。
- 修改任意销售记录。
- 配置卡酬规则。
- 查看日榜、总榜、结算数据。
- 导出 Excel。

## 5. 核心业务流程

### 5.1 成员录入流程
1. 成员登录系统。
2. 打开“今日录入”页面。
3. 选择日期。
4. 填写 40 套餐数量、60 套餐数量、备注（可选）。
5. 提交后直接生效。

规则：
- 每人每天只允许一条销售记录。
- 若同日记录已存在，则进入编辑流程而不是重复新增。

### 5.2 管理员修正流程
1. 管理员进入后台。
2. 按成员或日期搜索销售记录。
3. 修改数量或备注。
4. 保存后立即生效。

### 5.3 日榜流程
1. 选择某一天。
2. 系统查询当天所有销售记录。
3. 按成员汇总 `count_40`、`count_60` 和总数。
4. 按总数降序生成排行榜。

### 5.4 总榜流程
1. 选择开始日期和结束日期。
2. 查询时间段内所有销售记录。
3. 按成员聚合统计 40 总数、60 总数和总卖卡数。
4. 排序并展示。
5. 支持导出 Excel。

### 5.5 结算流程
1. 管理员选择结算时间段。
2. 系统查询该时间段内销售记录。
3. 按销售日期匹配对应生效中的卡酬规则。
4. 分别计算 40 套餐与 60 套餐应结金额。
5. 汇总每个成员的总应结金额。
6. 展示并支持导出 Excel。

## 6. 数据模型

### 6.1 `users`
用途：存储成员信息与角色。

建议字段：
- `id`
- `name`
- `phone` 或登录标识
- `username`
- `password_hash`
- `role`（member / admin）
- `status`
- `created_at`
- `updated_at`

### 6.2 `sales_records`
用途：存储每日销售记录。

建议字段：
- `id`
- `user_id`
- `sale_date`
- `count_40`
- `count_60`
- `remark`
- `created_at`
- `updated_at`
- `updated_by`（可选，建议预留）

约束：
- `user_id + sale_date` 唯一。

### 6.3 `commission_rules`
用途：存储卡酬规则。

建议字段：
- `id`
- `user_id`
- `price_40`
- `price_60`
- `effective_start`
- `effective_end`
- `created_at`
- `updated_at`

规则：
- 同一成员的规则时间段不允许重叠。

### 6.4 `settlement_snapshots`（可延期）
用途：存储某次结算快照，用于留档和不可变复核。

建议字段：
- `id`
- `user_id`
- `start_date`
- `end_date`
- `amount`
- `generated_at`

说明：
- 第一版可以不落表，先实时计算。
- 若后续需要“结算留档不可变”，再加入该表。

## 9. 异常处理与边界规则

### 9.1 重复提交
- 使用数据库唯一约束拦截重复提交。
- 前端可提前提示，后端必须兜底。

### 9.2 非法输入
- 数量只能为大于等于 0 的整数。
- 可设置合理上限。
- 后端必须再次校验。

### 9.3 卡酬规则缺失
- 结算时若找不到对应日期的规则，必须显式标记为“规则缺失”。
- 不允许默认按 0 元处理。

### 9.4 卡酬规则冲突
- 同一成员的生效时间段不可重叠。
- 保存规则时必须检查并拒绝冲突数据。

### 9.5 导出异常
- 无数据时提示为空或导出空模板。
- 导出失败要返回明确错误信息。

### 9.6 权限异常
- 未登录用户跳转登录页。
- 非管理员访问管理接口时必须拒绝。
- 后端权限校验不能仅依赖前端隐藏按钮。

## 10. 技术架构理解

整体架构：

```text
浏览器 / 手机
   ↓
Next.js 页面
   ↓
Next.js 服务端逻辑（Route Handlers / Server Actions）
   ↓
PostgreSQL 数据库
```

解释：
- Vercel 负责托管和运行网站程序。
- PostgreSQL 负责长期保存业务数据。
- Prisma 负责让服务端代码更方便地操作数据库。
- 排行榜和结算由服务端基于数据库实时计算。
- Excel 由服务端根据数据库查询结果动态生成。

## 13. 最终结论

该系统第一版采用响应式网页方案，技术栈为 Next.js + TypeScript + Prisma + PostgreSQL + ExcelJS，部署到 Vercel，域名继续使用 Cloudflare 管理。系统围绕成员录入、排行榜、卡酬规则、自动结算和 Excel 导出构建，优先追求尽快可用、长期持久化与后续低维护成本。
