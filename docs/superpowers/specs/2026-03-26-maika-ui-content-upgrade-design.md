# Maika UI 与内容系统升级设计

## 1. 目标

在不改动现有核心业务规则的前提下，对 Maika 做一次完整的前端产品化升级，覆盖以下目标：

- 将当前偏基础后台模板的界面升级为更现代、统一、具有团队目标感的产品界面
- 优先改善全站导航与移动端体验
- 其次改善表单、后台操作、筛选、保存反馈与表格可读性
- 再补强登录页氛围、榜单目标感、空态与关键状态呈现
- 新增两类登录后全站可见的内容能力：
  - 横幅一言
  - 全体公告

本次升级遵循既有 MVP 边界，不扩展通知中心、复杂图表、富文本系统或新的业务流程。

## 2. 设计方向

### 2.1 视觉主方向

采用 `C 为主，吸收 A 的动效` 的方案：

- 主基调偏“团队作战台 / 目标感 / 排名感”
- 色彩以深青蓝、青绿、浅青蓝渐变为核心
- 用局部暖光或高亮点缀增强层次，但不走艳俗风格
- 卡片层使用轻玻璃感、柔和阴影和大圆角
- 动效以轻量 CSS 过渡、渐入、位移、hover 光晕为主，不引入重型动画系统

### 2.2 产品气质

- 成员端：更聚焦、更有反馈感、更像“每日行动面板”
- 榜单页：更有竞争氛围和目标感
- 管理后台：更专业、更清晰、更适合长期使用

## 4. 全站壳层与导航

### 4.1 桌面端

- 将现有顶部导航升级为左侧主导航 + 右侧内容区
- 左侧导航分为两个区块：
  - 成员区
  - 管理区
- 当前页用更强的激活态高亮
- 壳层顶部保留身份信息与快捷动作位置

### 4.2 移动端

- 使用顶部栏 + 抽屉菜单
- 不做底部 Tab，因为管理员页面过多，不适合压缩成底部导航
- 抽屉中保留分组导航、当前页高亮、角色信息

### 4.3 页面头部结构

所有业务页统一采用以下头部结构：

- 页面标签
- 页面标题
- 一句解释
- 可选的状态卡 / 统计卡 / 快捷动作

目标是避免页面进入后只有表单或只有表格，增强定位感与完整度。

## 5. 组件设计

### 5.1 统一视觉基础

需要抽离统一样式与基础组件语义，用于以下元素：

- 页面容器
- Hero / Page Header
- 卡片
- 统计卡
- 按钮
- 输入框
- 选择框
- 文本域
- 状态标签
- 空态面板
- 表格容器
- 筛选工具条

优先使用 Tailwind + 项目内复用组件，不引入重型 UI 库。

### 5.2 动效规则

- 页面主体进入时做轻微淡入上移
- 卡片 hover 做阴影和边框高亮变化
- 按钮 hover / active / pending 有统一过渡
- 抽屉菜单开合采用短时长平滑过渡
- 横幅区域可做轻量轮播或淡入切换

不做长时间动画、复杂视差、3D 或明显拖慢交互的效果。

## 6. 页面级设计

### 6.1 登录页

目标：

- 增强品牌感和第一眼质感
- 保持登录动作简单直接

设计：

- 使用深色渐变背景 + 柔和高光
- 左侧为产品说明和能力标签
- 右侧为玻璃感登录卡
- 错误提示仍需清晰，但不破坏整体视觉

### 6.2 今日录入页

目标：

- 提升录入专注度
- 明确“同日覆盖”的业务反馈

设计：

- 顶部加入轻量状态卡
- 表单主体更聚焦，说明区单独放到辅助卡片中
- 成功、失败、覆盖提示采用更明确的状态样式
- 移动端保持单列，桌面端为主表单 + 说明侧栏

### 6.3 我的记录页

目标：

- 提升历史记录可读性

设计：

- 页面头部补充记录说明
- 表格样式升级，增加二级信息层次
- 空态除“暂无记录”外，给出建议动作

### 6.4 榜单页

目标：

- 提升团队竞争感和榜单识别度

设计：

- 页面头部加入统计卡
- Top 3 使用更强视觉强调
- 筛选表单与榜单结果分区更清楚
- 管理员导出入口更明确
- 无数据时提供切换日期或返回默认日期的建议

### 6.5 管理员首页

目标：

- 将当前占位页升级为可用的后台总览入口

设计：

- 提供后台模块快捷入口卡片
- 显示当前内容系统、销售管理、结算管理的跳转入口
- 可加入简短状态提示，而不做真实统计大盘

### 6.6 成员管理页

目标：

- 提升新增成员和成员维护的清晰度

设计：

- 左侧表单、右侧列表保持，但整体卡片层次升级
- 表单消息态统一
- 表格行内状态、角色标识、密码重置区域更清晰

### 6.7 销售记录管理页

目标：

- 提升筛选效率和保存反馈

设计：

- 筛选区独立成工具条
- 增加清空筛选入口
- 每条记录卡片化展示，强化“未保存 / 已保存 / 需补充”状态
- 输入和保存动作更聚焦

### 6.8 卡酬规则页

目标：

- 提高规则编辑与规则列表可读性

设计：

- 表单与列表并列，但表单信息密度更友好
- 规则时间区间显示更清楚
- 冲突或错误反馈保持突出

### 6.9 结算页

目标：

- 更清楚地展示“规则完整 / 规则缺失”

设计：

- 顶部保留时间范围和导出入口
- 若有缺失规则，在头部或结果区给出明显提醒
- 表格中统一状态标签和说明文案

## 7. 新增内容系统

本次新增两个独立系统，不做统一内容中心。

### 7.1 横幅一言

定位：

- 登录后全站顶部可见的短文案区域
- 用于营造氛围、做轻提醒或展示“一言”

规则：

- 登录页不显示
- 任意时刻前台只展示 1 条
- 内容池中可以维护多条
- 支持两种前台模式：
  - 随机显示
  - 自动轮播
- 内容来源支持：
  - 内置一言候选
  - 管理员自定义新增

后台能力：

- 新增横幅管理入口
- 管理员可新增、启用、停用、删除内容
- 管理员可切换展示模式：随机 / 轮播
- 可设置是否优先展示内置内容或自定义内容

建议字段：

- `id`
- `content`
- `author`（可选）
- `sourceType`：`BUILTIN` / `CUSTOM`
- `status`：`ACTIVE` / `INACTIVE`
- `createdAt`
- `updatedAt`

单独配置建议：

- `displayMode`：`RANDOM` / `ROTATE`
- `isEnabled`

### 7.2 全体公告

定位：

- 面向全体登录用户的正式通知区

规则：

- 登录后全站可见
- 登录页不显示
- 支持多条公告
- 支持置顶
- 支持发布时间
- 支持过期时间
- 已过期公告不在前台显示
- 前台优先展示置顶公告，其次按发布时间倒序

第一版不做：

- 已读未读
- 富文本编辑器
- 附件系统

后台能力：

- 新增公告管理入口
- 支持新增、编辑、启用、停用、置顶、设置发布时间和过期时间

建议字段：

- `id`
- `title`
- `content`
- `isPinned`
- `status`：`ACTIVE` / `INACTIVE`
- `publishAt`
- `expireAt`（可选）
- `createdAt`
- `updatedAt`

## 8. 前台展示规则

### 8.1 内容优先级

登录后页面顶部建议展示顺序：

1. 横幅一言
2. 公告区
3. 页面头部
4. 页面主体

### 8.2 页面接入策略

- `AppShell` 负责统一加载横幅与公告摘要
- 所有登录后页面自动接入
- 公共榜单页面若用户已登录，则使用壳层展示
- 未登录访问榜单页时，不显示横幅和公告

## 9. 数据流

### 9.1 视觉改造部分

- 不改变现有业务服务层接口
- 主要改造组件结构、页面布局和样式层

### 9.2 横幅与公告

- 新增 Prisma 模型
- 新增服务层：
  - 横幅内容查询
  - 当前横幅选择逻辑
  - 公告查询
  - 管理端增删改查
- 登录后壳层统一查询可见内容

## 10. 错误处理

需要统一以下状态样式与文案：

- 成功
- 错误
- 警告
- 空数据
- 缺失规则
- 已过期 / 未发布

内容系统需额外处理：

- 没有可用横幅时不报错，直接不显示
- 没有公告时显示简洁空态或不显示公告模块
- 公告发布时间或过期时间非法时后台拒绝保存

## 11. 测试与验证

本次改造完成后，至少验证以下内容：

### 11.1 基础验证

- `npm run lint`
- `npx tsc --noEmit`
- `npm run test`
- `npm run test:e2e`
- `npm run build`

### 11.2 关键交互验证

- 登录页可正常登录
- 成员端移动端导航可正常打开与切换
- 今日录入正常保存，成功 / 失败反馈正确
- 榜单页筛选与导出入口正常
- 管理后台所有页面可正常进入
- 横幅内容按规则显示
- 公告按置顶、发布时间、过期时间正确显示

### 11.3 视觉验证

- 桌面端与移动端无明显布局破坏
- 表格、卡片、输入控件视觉风格统一
- 动效不过重，不影响输入和导航

## 12. 实施顺序

建议按以下顺序实施：

1. 全局样式变量、字体、背景和壳层重构
2. 桌面端导航与移动端抽屉导航
3. 页面头部与通用卡片、按钮、表单、状态样式
4. 登录页、成员页、榜单页视觉升级
5. 管理后台四个页面视觉与交互升级
6. 横幅一言系统
7. 全体公告系统
8. 全量验证与 README / handoff 更新

## 13. 非目标

本次明确不做：

- 后端核心业务规则调整
- 图表大屏
- 消息通知中心
- 富文本公告编辑器
- 已读未读系统
- 外部一言 API 接入
- 重型前端组件库替换

## 14. 结论

本次升级是一次“保持业务逻辑稳定的前端产品化升级”，重点不在增加复杂业务，而在于：

- 让全站导航更顺
- 让成员和管理员操作更清晰
- 让界面更现代、统一、完整
- 让横幅一言与公告成为团队协同和氛围表达的一部分

如果该设计确认通过，下一步进入实现计划，并按壳层优先、内容系统后接入的顺序执行。
