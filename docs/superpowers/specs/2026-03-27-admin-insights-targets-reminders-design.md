# 管理员诊断中心、成员日目标与站内提醒设计

日期：2026-03-27
状态：已确认设计，待进入 implementation plan

## 1. 背景

当前项目已经具备：

- 成员每日录入
- 管理员销售审核
- 日榜 / 总榜
- 结算
- 公告与横幅
- 每日节奏摘要、临时前三 / 正式前三

现阶段的主要短板不是“数据有没有”，而是“数据能不能帮助管理员提前发现问题并推动动作”。管理员现在更多是在审核页和榜单页被动查看结果，缺少一个围绕成员状态、目标偏差和异常行为的主动诊断入口。成员端也缺少“今天该做到什么、离目标差多少、最近是否被提醒”的直接反馈。

## 2. 目标

第一期新增一个以管理员为主、成员同步受益的数据洞察模块，形成以下闭环：

1. 系统每天按成员自动生成今日目标建议
2. 管理员在统一的诊断面板中发现异常成员
3. 管理员可直接调整目标并发送站内提醒
4. 成员在 `/entry` 看到目标、达成差距、自我趋势和最近提醒

## 3. 非目标

第一期不做：

- 短信 / 微信 / 企业微信触达
- 独立消息中心
- 复杂审批流或多人协同跟进
- 黑盒型智能评分
- 成员公开对比他人姓名或完整排名策略
- 自动定时批量发送提醒

## 4. 用户与核心问题

### 4.1 管理员

核心问题：

- 今天谁最值得优先关注
- 这个成员的问题是结果异常、行为异常，还是两者都有
- 今日目标是否合理，是否需要微调
- 是否该立即提醒

### 4.2 成员

核心问题：

- 今天的目标是多少
- 当前离目标还差多少
- 和过去自己的状态比是在上升还是下降
- 最近有没有管理员提醒，需要优先处理什么

## 5. 方案概览

采用独立模块方案，在现有后台之外新增一个管理员主模块：

- 新页面：`/admin/insights`
- 保持现有 `/admin/sales` 为审核与记录处理页
- 在现有 `/entry` 上补成员侧目标与提醒反馈

结构上形成：

- 诊断：`/admin/insights`
- 执行：`/admin/sales`
- 接收反馈：`/entry`

## 6. 页面与信息架构

### 6.1 `/admin/insights`

这是管理员每天开工后的第一入口。

页面分四个区块：

1. 顶部总览区
   - 今日高风险成员数
   - 今日中风险成员数
   - 今日目标达成率
   - 今日已发送提醒数

2. 异常分布区
   - 风险分层统计
   - 异常原因分布
   - 原因类别首期固定为：
     - 结果下滑
     - 低于个人常态
     - 晚交 / 未交
     - 驳回频繁
     - 目标偏离过大

3. 异常成员诊断列表
   - 按风险高到低排序
   - 每个成员卡片展示：
     - 成员名
     - 风险等级
     - 今日目标 / 当前完成 / 差值
     - 最近几天趋势摘要
     - 异常原因标签
     - 建议动作
   - 卡片直接提供两个操作：
     - 调整今日目标
     - 发送站内提醒

4. 已处理 / 观察中区
   - 用于收纳已提醒或暂时观察的成员
   - 第一版可以只支持“待处理 / 已提醒 / 继续观察”三种状态

### 6.2 `/admin/sales`

继续承担：

- 审核销售记录
- 修正销售数量
- 处理历史日期记录

`/admin/insights` 发现问题后，如果管理员需要查看原始销售记录或处理驳回 / 审核，再跳转这里。

### 6.3 `/entry`

在现有录入页顶部补三块反馈：

1. 今日目标达成卡
   - 今日目标
   - 当前完成
   - 差距
   - 达成状态

2. 自我趋势反馈
   - 与过去自己的近几天常态对比
   - 输出上升 / 持平 / 下降

3. 最近提醒列表
   - 最近几条管理员站内提醒
   - 时间
   - 类型
   - 内容摘要

## 7. 数据与规则设计

### 7.1 今日目标

新增“按成员、按日期”的目标记录。

建议字段：

- `userId`
- `targetDate`
- `suggestedTotal`
- `finalTotal`
- `suggestionReason`
- `adjustedBy`
- `adjustedAt`

规则：

- 每天自动为每个成员生成建议目标
- 自动生成后即作为当天生效目标
- 首次生成时：
  - `finalTotal = suggestedTotal`
- 管理员可修改最终目标
- 若管理员当天多次调整，以最后一次生效值为准

### 7.2 异常评分

采用可解释的混合评分，不做黑盒算法。

评分信号分三组：

1. 结果信号
   - 今日完成明显低于个人近 7 天常态
   - 今日距离目标差距过大

2. 行为信号
   - 晚交
   - 未交
   - 近期驳回偏多

3. 对比信号
   - 相比自己最近几天显著下滑

输出结果包括：

- 风险分值
- 风险等级：`HIGH / MEDIUM / LOW`
- 原因拆解列表
- 建议动作

建议动作首期由规则映射产出，例如：

- 目标落后明显：建议“发送目标落后提醒”
- 连续低迷：建议“调整今日目标后提醒”
- 晚交 / 未交：建议“发送补交提醒”
- 驳回频繁：建议“先去审核页复核原始记录”

### 7.3 目标建议算法

目标建议以成员个人近期表现为基础，不做统一目标。

建议算法：

1. 先取成员近几天真实销量作为基础值
2. 再结合最近状态做微调

优先依据：

- 近几天销量
- 迟交 / 漏交情况
- 驳回情况

结果示例：

- 状态稳定且上升：建议略高于近几天均值
- 状态不稳：建议接近均值
- 连续下滑或被驳回较多：建议更保守

目标建议必须展示“建议原因”，不能只给数字。

### 7.4 站内提醒

新增提醒记录。

建议字段：

- `id`
- `userId`
- `type`
- `title`
- `content`
- `status`
  - `UNREAD`
  - `READ`
- `sentBy`
- `sentAt`

提醒发送方式：

- 模板
- 模板后微调
- 完全自定义

首期模板类型：

- 目标落后提醒
- 迟交 / 未交提醒
- 关注状态提醒
- 管理员自定义提醒

## 8. 关键边界条件

- 成员当天未提交记录时，目标卡仍显示目标与当前 `0`
- 成员被驳回后重提，不重置目标与提醒记录
- 多次提醒按时间倒序展示
- 多次目标调整以最后生效值为准，但建议保留调整记录
- 管理员诊断页与审核页职责分离，不在 `insights` 内直接处理销售审核

## 9. 第一期开启范围

第一期包含：

- `/admin/insights` 新页
- 顶部总览
- 异常成员诊断列表
- 今日目标建议与人工调整
- 站内提醒发送
- `/entry` 顶部目标达成卡
- `/entry` 自我趋势反馈
- `/entry` 最近提醒列表
- 所需数据模型、服务层和测试

第一期不包含：

- 复杂消息中心
- 外部消息通道
- 自动批量调度
- 成员公开对比别人

## 10. 实施顺序

建议按以下顺序实施：

1. 数据模型与服务层
   - 成员日目标
   - 异常评分
   - 站内提醒

2. `/admin/insights` 只读版
   - 顶部总览
   - 异常诊断列表

3. 接目标调整能力

4. 接站内提醒发送能力

5. 补 `/entry` 目标 / 趋势 / 提醒反馈

6. 补图表、排序、稳定性与回归测试

## 11. 测试策略

需要重点验证：

- 目标建议计算正确
- 异常评分与原因拆解可重复、可解释
- 管理员能从诊断页直接调整目标
- 管理员能发送站内提醒
- 成员端正确显示目标、完成、差距、趋势和提醒
- 历史日期、未提交、被驳回、重新提交不会打坏现有审核流

## 12. 风险与后续扩展

主要风险：

- 评分规则过多会让第一版难以理解
- 管理员动作过重会让 `insights` 重新变成复杂后台
- 外部 Neon 当前仍未补 Prisma baseline，未来迁移管理需单独处理

后续自然扩展方向：

- 更强的图表分析
- 批量提醒
- 成员分层与分组
- 自动提醒策略
- 更细的运营跟进状态

## 13. 推荐结论

推荐按本设计进入 implementation plan。

理由：

- 管理员优先受益，符合当前优先级
- 同时给成员明确目标与提醒反馈，形成闭环
- 不破坏现有审核页
- 首期范围完整，但仍可分层落地
