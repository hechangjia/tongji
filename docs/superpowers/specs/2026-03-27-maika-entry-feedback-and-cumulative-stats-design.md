# Maika 录入反馈与累计统计体验增强设计

## 1. 背景

当前 Maika 已完成一轮 UI 与内容系统升级，但从产品体验角度看，第一优先问题仍集中在两个高价值场景：

1. `今日录入页` 在保存成功后，用户仍可能不够确定“刚刚到底保存了什么”
2. 系统缺少一个足够直观的累计结果表达层，无法同时满足成员端的参与感和管理员端的观察需求

这份设计聚焦于第一轮体验增强，只解决以上两个问题，不扩展到其他页面的大范围改版。

## 2. 目标

本轮增强目标：

- 让成员在 `/entry` 完成保存后，能在 2 秒内确认本次提交内容和业务结果
- 让成员端能更直观看到“当前累计表现”
- 让管理员端能更方便地观察“成员累计买卡趋势”
- 在不改变现有业务规则的前提下，增加更清晰、更美观的结果表达层

## 3. 范围

### 3.1 本次范围

- `/entry` 页面保存成功后的确认反馈升级
- 成员端 `/leaderboard/range` 页面中的累计买卡统计模块
- 管理员首页中的累计买卡趋势模块
- 图表相关的空态、无数据态、口径切换态设计

### 3.2 明确不在本次范围

- 登录页进一步品牌化升级
- 全站统一状态系统的全面重构
- `我的记录` 页面复盘能力增强
- 新增独立统计页
- 金额统计图、复杂多指标联动图
- 自定义成员多选、复杂筛选器、导出能力扩展
- 新增数据库字段或新的核心业务规则

## 4. 关键约束

- 保持现有业务规则不变
- 角色仍为 `MEMBER / ADMIN`
- 套餐仍为 `40 / 60`
- 每人每天仍只有一条销售记录
- 统计必须基于现有销售记录推导，不新增新的录入口径
- 第一版优先保证“默认好懂、默认好看、默认不乱”，不引入复杂分析交互

## 5. 统计口径定义

本次新增的“累计买卡数量”定义为：

- 在给定时间范围内，满足筛选条件的销售记录条数
- `总量` = `40` 套餐记录条数 + `60` 套餐记录条数
- `40` / `60` 拆分统计 = 分别统计对应套餐的记录条数

该定义的意义是：

- 不依赖新的数量字段
- 可直接复用现有销售记录
- 对成员和管理员都容易理解

本次不引入金额累计与数量累计混合展示，避免第一版图表目标失焦。

## 6. 设计一：今日录入页保存确认反馈

### 6.1 要解决的问题

当前录入页在保存后需要给出更强的“结果确认感”。用户不应只知道“保存成功了”，而应清楚知道：

- 保存的是哪一天的数据
- 保存了哪种套餐
- 销售额是多少
- 备注内容是否正确
- 这次是新增还是覆盖

### 6.2 核心方案

`/entry` 在保存成功后，不跳页、不切换成整页完成态，而是在页面头部下方展示一张高优先级的结构化成功卡片。

这张卡片是页面当前最重要的反馈层，职责只有一个：

`让用户快速确认本次提交内容和业务结果。`

### 6.3 成功卡片结构

成功卡片固定包含：

- 主标题
- 一句解释文案
- 关键摘要字段
- 后续动作

推荐字段：

- 销售日期
- 套餐
- 销售额
- 备注摘要
- 保存时间

推荐动作：

- 主动作：`查看我的记录`
- 次动作：`继续调整今天记录`

这里不把“再次保存”作为主动作，因为保存后的第一自然动作应是确认结果已落库，而不是立即重提。

### 6.4 状态分支

成功卡片至少区分以下 3 种业务结果：

#### 普通保存成功

- 主标题：`今日记录已保存`
- 副文案：强调“以下为本次提交内容”

#### 覆盖保存成功

- 主标题：`今日记录已更新`
- 副文案：明确说明系统已用本次内容覆盖今天原有记录
- 卡片内部追加 `已覆盖旧记录` 的状态标识

#### 保存失败后重试成功

- 在成功卡片中增加一条恢复性提示
- 明确告诉用户“刚才的提交未成功，这次已经保存完成”

### 6.5 加载与失败状态

第一版不应只依赖按钮 pending 态。

在提交中：

- 页面头部状态区进入轻量忙碌态
- 文案明确表示“正在提交今日记录”

在提交失败时：

- 错误提示必须明确原因或至少说明失败
- 需要保留继续修改并重试的路径
- 不应让成功提示、表单消息和 toast 同时抢占主反馈位

### 6.6 视觉与排版

推荐排版：

- 第一行：状态标签 + 主标题
- 第二行：一句解释文案
- 第三行：摘要字段区
- 第四行：动作按钮

设计原则：

- 卡片比普通内容卡更醒目，但不做系统公告式整条横幅
- 备注字段最多展示 1 到 2 行，避免卡片过厚
- 移动端使用单列字段布局
- 页面主反馈永远只有一个视觉中心，避免信息打架

## 7. 设计二：累计买卡统计模块

### 7.1 角色分工

同一套数据主题使用两种展示方式：

- 成员端：强调直观、参与感、排名感
- 管理员端：强调趋势观察和状态判断

这意味着第一版不采用“同一个图表组件原样复用到所有页面”的方案，而是按角色做不同表达。

### 7.2 放置位置

成员端：

- 第一版放在 `/leaderboard/range`
- 不放进 `/entry`

原因：

- `/entry` 当前第一目标是“录入信心”，不应被统计图分散注意力
- `累计买卡` 本质是区间统计，更适合挂在现有区间榜单页，而不是日榜页

管理员端：

- 放在 `/admin` 首页

原因：

- 这是最合适的总览位置
- 也符合“具体细节由管理员在后台操作”的要求

### 7.3 默认时间范围与可切换项

默认展示：

- `本月`
- `总量`

管理员可切换：

- 时间范围：`本月 / 近 30 天 / 全历史`
- 统计口径：`总量 / 40 / 60`

成员端第一版不提供复杂切换器，保持默认表达稳定、简洁、可快速理解。

## 8. 成员端视图设计

### 8.1 核心表达方式

成员端默认使用：

`累计排名条形图 + 当前用户高亮`

模块包含：

- 标题：`本月累计买卡`
- 一句轻说明：`按全体成员累计数量统计`
- 横向条形排行列表

第一版默认以“本月”为初始区间，并挂接在现有 `/leaderboard/range` 语境下；如果成员需要看其他时间范围，应复用该页面已有的区间筛选能力，而不是为图表单独增加复杂控制器。

每行至少显示：

- 排名
- 成员名
- 累计数量

当前登录用户所在行需要明显高亮，即使不在前 3 也必须容易找到。

### 8.2 设计目标

成员端图表的目标不是分析，而是让用户快速回答两个问题：

- 谁当前累计更多
- 我离前面的人差多少

因此不在成员端第一版使用复杂趋势图。

### 8.3 成员端边界状态

- 当前时间范围内无数据时，显示简洁空态
- 空态文案需提示“可调整当前区间后查看”，复用现有区间榜单筛选入口
- 如果当前用户在榜单靠后，也必须仍然能看见自己的位置

## 9. 管理员端视图设计

### 9.1 核心表达方式

管理员端默认使用：

`累计趋势折线图 + 顶部切换器 + 简洁排行摘要`

模块包含：

- 标题：`成员累计买卡趋势`
- 控件区：
  - 时间范围切换
  - 统计口径切换
- 主图：累计趋势折线图
- 图下：当前区间 Top 成员摘要

### 9.2 显示规则

为避免图表过乱，第一版采用限制策略：

- 默认只展示 Top 5 成员曲线
- 其他成员不在主图中同时展开
- 其余成员通过排行摘要承接

这样可以保证图表仍然清晰、美观、可读。

### 9.3 管理员端交互原则

管理员端允许控制细节，但第一版只开放必要控制项：

- 时间范围切换
- 统计口径切换

第一版不开放：

- 自定义多成员勾选
- 多图联动
- 数量/金额混合对比
- 独立配置页

这些需求若后续确有价值，再进入下一轮设计。

## 10. 图表状态与边界设计

第一版至少覆盖以下状态：

### 10.1 正常有数据

- 成员端展示累计排名条形图
- 管理员端展示趋势图和 Top 摘要

### 10.2 当前筛选范围无数据

- 不留白，不伪装成系统故障
- 明确显示“当前时间范围内暂无累计买卡数据”
- 管理员端额外强调“这是当前筛选条件下无数据”

### 10.3 成员数量较多

- 管理员趋势图不同时绘制全部成员
- 默认限制主图展示数量，保持图面干净

### 10.4 切换到 `40` 或 `60` 后部分成员为 0

- 视为正常业务结果，不视为异常
- 图表与摘要应自然容纳低值或空值

### 10.5 当前用户不在前列

- 成员端仍需明确标出“我”的位置
- 不能让成员端图表变成“只有别人值得看”

## 11. 实施建议顺序

建议本轮体验增强按以下顺序实施：

1. 完成 `/entry` 成功卡与状态反馈升级
2. 在成员端榜单页接入累计排名条形图
3. 在管理员首页接入累计趋势图与切换器
4. 补齐图表空态、低值态和高成员数场景验证

这个顺序的原因是：

- 录入反馈直接影响最高频主流程，优先级最高
- 累计统计模块属于结果表达层，适合作为第二步增强

## 12. 验收标准

本轮设计通过后，至少应满足：

- 用户在 `/entry` 保存后，能明确确认本次提交的关键内容
- 用户能区分“保存成功”与“覆盖更新”
- 成员端能在榜单页看到直观的累计结果，并快速定位自己
- 管理员端能在首页查看累计趋势，并切换时间范围与统计口径
- 图表在无数据、成员较多、单套餐数据偏少时仍保持可理解
- 本轮增强不改变现有业务规则和数据录入口径

## 13. 延后项

以下增强明确留到后续轮次，不在本次实现：

- 登录页继续打磨
- 全站统一状态系统重构
- `我的记录` 复盘图表
- 金额统计图
- 更复杂的运营看板
- 更丰富的管理员统计配置

## 14. 结论

本轮体验增强的主题应明确为：

`把 Maika 从“能录入”升级成“让人敢确认、愿意回看、知道下一步”的产品。`

第一版最值得做的两个动作是：

1. 用结构化成功卡片提升 `/entry` 的保存确认感
2. 用双层表达的累计统计模块，同时满足成员端直观感和管理员端观察需求

如果该设计确认通过，下一步进入 implementation plan，范围仅围绕本 spec 展开，不再把其他体验想法混入第一轮交付。
