# Maika 每日节奏、审核状态与前三单激励设计

## 1. 背景

Maika 已完成第一轮体验增强：

- `/entry` 具备结构化保存确认卡
- `/leaderboard/range` 已有累计买卡排行
- `/admin` 已有累计趋势摘要

但当前产品仍缺少一层更强的“每日节奏感”和“状态驱动感”：

- 成员登录后虽然能录入和看榜单，但不够清楚“我现在处于什么状态、下一步该做什么”
- 管理员虽然能看趋势和列表，但缺少一层围绕“今日任务”的摘要与入口
- 项目中已存在“前三单激励”的真实业务动机，但系统还没有支持“按提交时序 + 管理员审核”来表达这一激励

本轮设计目标，是在现有销售记录流程上加一层轻量节奏与审核机制，而不是额外创造一套新工单系统。

## 2. 目标

本轮增强目标：

- 让成员登录后能快速知道：
  - 今天是否已提交
  - 当前审核状态
  - 自己是否处于临时前三 / 正式前三
  - 当前最推荐的下一步动作
- 让管理员登录后能快速知道：
  - 今天是否已有成员提交
  - 当前待审核记录数量
  - 今日前三单是否已确认
  - 当前最推荐的管理入口
- 在现有销售记录模型上支持“秒级提交时间 + 审核状态 + 临时/正式前三”的流程
- 为后续可能加入的截图上传与审核依据预留清晰挂点，但本轮不实现文件上传

## 3. 范围

### 3.1 本次范围

- 成员端 `/entry` 的状态摘要区
- 管理员端 `/admin` 的状态摘要区
- 管理员端 `/admin/sales` 的审核状态与按时间排序
- 每日“临时前三 / 正式前三”规则与结果展示
- 销售记录上的秒级最后提交时间、审核状态与审核信息

### 3.2 明确不在本次范围

- 订单完成截图上传
- 文件存储或媒体服务
- 独立“收单”模块
- 激励金额或结算联动
- 多级审批流
- 激励发放记录
- 排名历史版本追踪

## 4. 核心原则

- 不创建新业务主表，优先在现有销售记录流程上扩展
- 临时前三和正式前三都应是派生结果，不额外写死排名字段
- 排名依据优先围绕“成员最后一次提交完成时间”
- 管理员拥有正式确认权
- 成员端看到的是“当前结果和下一步”，而不是后台全量审批细节

## 4.1 业务日口径

本轮所有“今日 / 当天”相关逻辑统一按业务时区处理：

- 业务时区：`Asia/Shanghai`
- 日界线：当天 `00:00:00` 到当天 `23:59:59.999`
- “今日记录”
- “今日是否有提交”
- “今日待审核数量”
- “今日临时前三 / 正式前三”

都必须基于同一业务日口径计算。

本轮不允许：

- 一部分逻辑按服务器默认时区
- 一部分逻辑按用户浏览器时区

否则会导致成员端与管理员端结果不一致。

## 5. 页面职责设计

### 5.1 成员端 `/entry`

这是成员每日动作的起点，因此它应成为“今日工作面板”。

这里需要展示：

- 今天是否已提交
- 本条记录最后提交时间，精确到秒
- 当前审核状态
- 是否处于临时前三
- 是否已进入正式前三
- 当前最推荐的下一步动作

成员端 `/entry` 不承担审核操作，但应承担：

- 提交后的状态反馈
- 审核状态与激励状态的轻量展示
- 登录后节奏入口

### 5.2 管理员端 `/admin`

管理员首页不承担完整审核流，但应承担“今日管理摘要”。

需要展示：

- 今日是否已有成员提交
- 今日待审核记录数量
- 今日前三单是否已确认
- 一条明确的状态结论
- 一个主推荐动作和两个次级入口

主推荐动作应优先把管理员带往：

- `/admin/sales`（审核今日记录）
- 或累计趋势区域（当今日数据已稳定时）

### 5.3 管理员端 `/admin/sales`

这是本轮审核机制的主操作页面。

这里需要支持：

- 每条记录显示最后提交时间，精确到秒
- 按提交时间排序
- 显示当前审核状态
- 显示临时前三标识
- 显示正式前三标识
- 审核通过 / 驳回动作

这个页面承担：

- 审核决策
- 当日时序核对
- 前三单正式确认

### 5.4 成员端榜单相关页

榜单页承担结果展示，不承担审核操作。

建议展示：

- 今日正式前三
- 或临时前三但带明显“待审核”标识

目标是增强结果感，而不是复制后台审核逻辑。

## 6. 状态摘要与下一步入口

### 6.1 成员端摘要区

成员端摘要区采用：

`一句状态结论 + 一个主推荐动作 + 两个次级入口`

而不是更多指标卡。

优先回答：

- 我今天现在是什么状态
- 我下一步最该点哪里

建议成员端规则：

#### 状态 1：今天未提交

- 状态结论：`你今天还没有提交记录，先完成今日录入。`
- 主动作：`去录入今天数据`

#### 状态 2：今天已提交，且刚完成或刚更新

- 状态结论：`你今天的记录已完成，建议确认榜单和当前激励状态。`
- 主动作：`查看总榜`

#### 状态 3：今天已提交，等待审核

- 状态结论：`你今天已提交记录，当前正在等待管理员审核。`
- 主动作：`查看当前状态`

#### 状态 4：今天已通过审核

- 状态结论：`你今天的记录已通过审核，可以继续关注排名和本月累计。`
- 主动作：`查看本月累计`

次级入口建议固定在以下候选中选两项：

- `查看我的记录`
- `查看总榜`
- `查看公告`

### 6.2 管理员端摘要区

管理员端也采用：

`一句状态结论 + 一个主推荐动作 + 两个次级入口`

优先回答：

- 今天是否已有提交
- 待审核任务还有多少
- 现在最应该点哪里

建议管理员端规则：

#### 状态 1：今天暂无成员提交

- 状态结论：`今天还没有看到成员提交，建议先检查录入情况或发出提醒。`
- 主动作：`查看销售记录`

#### 状态 2：今天已有提交，且存在待审核

- 状态结论：`今天已有成员提交，当前还有记录待审核，建议优先确认今日前三。`
- 主动作：`去审核今日收单`

#### 状态 3：今天已有提交，且审核已基本完成

- 状态结论：`今天记录已基本确认，可以查看累计趋势和总榜变化。`
- 主动作：`查看累计趋势`

次级入口建议固定在以下候选中选两项：

- `管理公告`
- `管理横幅`
- `查看总榜`

## 7. 数据层扩展

### 7.1 建议字段

本轮建议在现有销售记录上增加以下信息：

这些状态字段的主数据来源建议固定为：

- 记录本身：销售记录表
- 成员角色与成员启用状态：现有用户表 / 权限体系中的 `role` 与 `status`

#### 最后提交时间

用途：

- 记录成员最后一次提交完成的时间，精确到秒
- 作为临时前三和正式前三的排序依据
- 在成员端和管理员端展示“本条记录最后提交于何时”

这个字段应保存完整时间精度；
前台显示可以精确到秒，但排序必须使用底层完整时间值，而不是显示后的字符串。

#### 审核状态

建议状态：

- `PENDING`
- `APPROVED`
- `REJECTED`

用途：

- 区分待审核、已通过、已驳回
- 支撑“临时前三”和“正式前三”的差异

#### 审核时间

用途：

- 记录管理员何时通过或驳回
- 后续接截图审核时可复用

#### 审核备注（可选）

用途：

- 驳回原因说明
- 留存复核上下文

### 7.2 明确不建议增加的字段

本轮不建议增加：

- `temporaryRank`
- `finalRank`

原因：

- 它们都属于派生结果
- 一旦成员再次提交或管理员再次审核，会带来额外一致性维护成本

本轮的正确方式应是：

- 临时前三 = 基于当日记录实时计算
- 正式前三 = 基于 `APPROVED` 记录实时计算

## 8. 前三单激励规则

### 8.1 排名依据

本轮前三单激励的排序主口径为：

`成员最后一次修改并提交完成的时间`

不采用：

- 首次提交时间
- 管理员审批通过时间
- 截图完成时间

原因：

- 你已明确成员可能在提交后继续修改
- 最终应以成员最后一次确认提交完成的时间作为参与排序的依据

### 8.1.1 “单”的实体定义

本轮“前三单”里的“单”不是独立新单据，而是：

`每个成员在同一业务日内唯一的一条销售记录`

这与项目现有业务规则保持一致：

- 每成员每天仅一条销售记录
- 同一天再次保存（即再次提交） = 更新同一条记录

因此：

- 一个成员在一天内不会占据多条“单”
- 当天前三的竞争单位是“成员当日记录”，不是“多次提交动作”

### 8.1.2 稳定排序规则

当天前三排序采用：

1. `lastSubmittedAt` 升序
2. 若 `lastSubmittedAt` 完全相同，则按 `salesRecord.id` 升序作为稳定 tie-break

这样可以保证：

- 成员端与管理员端排序一致
- 不会因为数据库返回顺序不同而导致前三不稳定

### 8.2 临时前三

临时前三的定义：

- 基于当天所有“有效参与竞争”的销售记录
- 按最后提交时间升序排序
- 取前 3 条

临时前三对成员端可见，但必须带有明确标识：

`临时前三，待管理员确认`

这里“有效参与竞争”的判定固定为：

- 记录属于当前业务日
- 记录对应成员角色为 `MEMBER`
- 记录对应成员状态为 `ACTIVE`
- 记录当前审核状态为 `PENDING` 或 `APPROVED`

明确排除：

- `REJECTED` 记录

这意味着：

- 一条记录一旦被管理员驳回，会立即移出临时前三竞争集合
- 只有成员再次修改并重新提交、状态回到 `PENDING` 后，才重新参与临时前三
- 本轮不额外要求“数量必须大于 0”才能参与竞争；只要记录能通过现有录入校验并处于有效状态，就视为一条有效提交

### 8.3 正式前三

正式前三的定义：

- 只从当天 `APPROVED` 记录中计算
- 仍按最后提交时间升序排序
- 取前 3 条

这样意味着：

- 临时前三和正式前三可能不同
- 被驳回的记录不能进入正式前三
- 后续通过的记录可向前补位

### 8.4 “今日前三单是否已确认”的语义

本轮不增加额外的“锁定确认”动作。

“今日前三单是否已确认”应理解为：

- 当前基于当天 `APPROVED` 记录计算出来的正式前三已经可展示
- 如果当天仍存在 `PENDING` 记录，则管理员摘要区应表达为“今日前三仍可能变化”
- 如果当天不存在 `PENDING` 记录，则管理员摘要区可表达为“今日前三已确认完成”

也就是说：

- “确认”是派生语义，不是额外状态字段
- 正式前三始终由 `APPROVED` 集合实时计算

管理员摘要区的边界判定建议固定如下：

#### 场景 1：当天无提交

- 结论：`今天还没有成员提交记录`
- 是否已确认：否
- 主动作：`查看销售记录`

#### 场景 2：当天有提交，但 `APPROVED` 数量为 `0`

- 结论：`今天已有成员提交，但前三尚未形成，建议优先审核`
- 是否已确认：否
- 主动作：`去审核今日收单`

#### 场景 3：当天有提交，`APPROVED` 数量为 `1` 或 `2`

- 结论：`今天已有部分记录通过审核，但正式前三尚未满 3 单`
- 是否已确认：否
- 主动作：`继续审核`

#### 场景 4：当天 `APPROVED` 数量已达到 `3`，但仍存在 `PENDING`

- 结论：`今天正式前三已形成，但仍有待审核记录，前三结果仍可能变化`
- 是否已确认：否
- 主动作：`继续审核`

#### 场景 5：当天不存在 `PENDING`，且 `APPROVED` 数量为 `3` 或以上

- 结论：`今天前三已确认完成`
- 是否已确认：是
- 主动作：`查看累计趋势`

#### 场景 6：当天所有提交均为 `REJECTED`

- 结论：`今天已有提交，但当前没有可进入正式前三的有效记录`
- 是否已确认：否
- 主动作：`查看销售记录`

### 8.5 当成员再次修改时

只要成员再次提交：

- 最后提交时间刷新
- 审核状态重置为 `PENDING`
- 临时前三实时重算
- 正式前三需等待管理员重新确认后再更新

这条规则是本轮流程的核心，必须明确。

## 9. 审核流程设计

### 9.1 成员提交

成员在 `/entry` 保存（即提交）后：

- 更新当天销售记录
- 更新最后提交时间
- 审核状态重置为 `PENDING`

如果该记录原本已是 `APPROVED` 或 `REJECTED`，再次提交后也必须重置为 `PENDING`。

### 9.2 管理员审核

管理员在 `/admin/sales` 对当日记录执行：

- 通过
- 驳回

通过后：

- 状态变为 `APPROVED`
- 写入审核时间
- 若该记录在已通过集合中排进前 3，则进入正式前三

驳回后：

- 状态变为 `REJECTED`
- 不参与正式前三
- 可选填写驳回备注

### 9.3 再次提交后的行为

若一条已通过或已驳回的记录被成员再次修改并提交：

- 状态重新回到 `PENDING`
- 需要管理员重新审核
- 临时前三实时变化
- 正式前三按审核结果重算

## 10. 成员端展示规则

成员端需要看见的状态包括：

- `待审核`
- `已通过`
- `已驳回`
- `当前处于临时前三`
- `已进入正式前三`

成员端不需要看到：

- 全量审核日志
- 后台审批规则细节
- 其他成员的审核细节

目标是：

- 增强激励感
- 增强节奏感
- 让成员清楚自己当前处于什么阶段

## 11. 管理员端展示规则

管理员端除了现有销售记录信息外，需要新增：

- 最后提交时间（秒级）
- 审核状态标签
- 临时前三标识
- 正式前三标识
- 审核动作

并支持：

- 默认仅看“今日”
- 默认采用“状态分组优先 + 分组内按最后提交时间排序”

目标是让管理员一眼分清：

- 哪些是今天最新提交的
- 哪些待审核
- 哪些属于前三单竞争区

管理员端 `/admin/sales` 默认排序规则建议固定为：

1. `PENDING`
2. `APPROVED`
3. `REJECTED`

每个状态分组内部再按以下顺序排序：

1. `lastSubmittedAt` 升序
2. `salesRecord.id` 升序

这样设计的原因是：

- 管理员最该先看的始终是待审核记录
- 同时又保留清楚的时间先后顺序
- 审核完成与驳回记录不会挡住当天待处理项

## 12. 历史数据迁移与回填策略

本轮若新增 `lastSubmittedAt` 字段，历史数据需要给出最小可行回填策略。

建议规则：

- 历史记录的 `lastSubmittedAt` 先用现有 `updatedAt` 回填
- 对历史记录不追求“真实首次提交时间”，只追求可排序的近似可用时间
- 上线后，新产生或新修改的记录使用真实提交时间

这意味着：

- 历史日期上的前三结果可能只能视为近似结果
- 新业务日上的前三结果才具备完整可信度

验收时应明确区分：

- 历史数据：允许近似
- 新提交数据：要求准确

## 13. 与现有第一轮体验增强的关系

这轮设计与前一轮体验增强不是独立关系，而是增强关系：

- `/entry` 的成功卡负责回答“你刚刚保存了什么”
- 本轮的状态摘要负责回答“你现在处于什么状态，下一步该去哪”
- 累计统计负责回答“你当前整体表现如何”
- 前三单审核与激励负责回答“你今天在时序激励里处于什么位置”

这四者合在一起，才会让 Maika 更像“每日工作面板”，而不是功能拼盘。

## 14. 实施建议顺序

建议按以下顺序实施：

1. 补数据字段与审核状态
2. 在 `/admin/sales` 做时间排序与审核动作
3. 在成员端 `/entry` 加入状态摘要与审核/激励提示
4. 在管理员端 `/admin` 加入管理摘要与推荐入口
5. 在榜单页补正式前三或临时前三结果展示

这样可以先把核心业务流走通，再补前台节奏感。

## 15. 结论

本轮最合理的方向不是新建“收单系统”，而是：

`在现有销售记录流程上增加秒级提交时间、审核状态、临时前三与正式前三机制，并把成员端 /entry 与管理员端 /admin 提升为真正的每日节奏入口。`
