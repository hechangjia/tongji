# 校园卡销售系统重构设计

## 1. 背景

当前项目已经具备以下基础能力：

- 管理员 / 成员账号登录
- 成员录入销售数据
- 管理员查看、修改、审核销售记录
- 日榜 / 总榜 / 结算 / 导出
- 佣金规则配置

但当前系统的核心业务模型仍然是：

- 只有 `ADMIN` / `MEMBER` 两类角色
- 没有小组、组长、组内权限边界
- 没有识别码库存、导入、分发、追踪
- 没有“预发放给新生”的业务层
- 结算口径建立在成员直接填写 `40 / 60` 套餐数量上

老板提出的新要求不是局部增量，而是一次业务核心升级：

- 系统要支持 `管理员 / 组长 / 组员` 三角色
- 识别码由管理员导入并分发给成员
- 成员围绕识别码完成预发放、售出和异常处理
- 看板和权限按小组分层
- 薪资口径继续保留 `40 / 60` 套餐规则

本设计采用 **渐进式重构** 方案：保留现有项目的认证、页面框架、后台导航、导出和结算基础设施，在此基础上替换业务主链路。

## 2. 设计目标

本次重构目标如下：

1. 在现有项目内升级成三角色系统：`ADMIN / LEADER / MEMBER`
2. 引入小组模型和组长权限边界
3. 引入识别码库存、导入、分发、预发放、售出闭环
4. 保留 `40 / 60` 套餐对薪资和佣金结算的核心作用
5. 让排行榜、明细、结算全部建立在“识别码售出 + 套餐类型”的统一事实源上

## 3. 本期范围

### 3.1 本期必须交付

- 登录页支持“登录 / 注册”切换
- 注册仅开放组员注册，输入账号密码即可
- 管理员创建小组、指定组长、维护成员备注
- 组长可维护自己小组的名称和口号
- 管理员上传识别码 Excel/CSV，前端解析并批量导入
- 管理员把识别码分发给成员，并保留完整分发记录
- 成员在自己名下识别码上登记预发放记录
- 成员在自己名下识别码上完成售出录单
- 每次成交必须绑定：
  - 一个识别码
  - 一个套餐类型：`40` 或 `60`
- 管理员可修改/撤销全平台销售记录
- 组长可修改/撤销自己组内销售记录
- 全平台 / 小组 / 组内 / 个人数据看板与排名
- 结算继续以 `40 / 60` 佣金规则为主口径

### 3.2 本期明确不做

- 手机号注册与真实短信验证码
- 组长自主分发识别码
- 成员自行从库存池提取识别码
- 新生 QQ Excel 完整导入与分发流程

### 3.3 本期只做预留

- 新生线索数据结构与后续入口预留
- 未来可接短信验证码 / 线索批量导入

## 4. 关键业务决策

以下规则已经锁定，后续实现不得偏离：

- 注册只允许普通成员注册
- 组长由管理员从成员中指定
- 管理员拥有全平台全部权限
- 组长拥有：
  - 全平台小组排名查看权限
  - 本组成员明细查看权限
  - 本组销售记录修改 / 撤销权限
- 组员只能操作自己名下识别码与自己的销售录单
- 识别码只允许管理员分发，不允许成员自提
- 管理员分发识别码时不受“每人最多 30 个”的限制
- 识别码存在预发放记录，状态固定为：
  - `PRE_DELIVERED`
  - `USED`
  - `UNUSED`
  - `TRANSFERRED`
- 本期保留 `40 / 60` 套餐为佣金与薪资结算主口径
- 每次识别码成交必须对应一个套餐类型
- 组长差额分润按 `40 / 60` 分开计算
- 旧的管理员审核流下线，不再作为新主流程
- 新业务录入提交后直接生效，组长/管理员负责事后修改或撤销

## 5. 角色与权限模型

### 5.1 管理员 `ADMIN`

可查看：

- 全平台所有小组
- 所有成员
- 所有识别码库存、分发、预发放、售出数据
- 所有排行榜、汇总、结算与导出

可操作：

- 创建 / 编辑 / 停用小组
- 指定 / 更换组长
- 调整成员所属小组、备注、状态
- 导入识别码
- 分发识别码
- 修改 / 撤销任意销售记录
- 维护佣金规则

### 5.2 组长 `LEADER`

可查看：

- 全平台小组排名
- 自己所在小组全部成员业绩、识别码、预发放、薪资明细
- 自己小组独立看板

可操作：

- 修改自己小组名称与口号
- 修改 / 撤销自己组内成员的销售记录

不可操作：

- 分发识别码
- 维护全平台佣金规则
- 管理其他组成员

### 5.3 组员 `MEMBER`

可查看：

- 自己名下识别码
- 自己的预发放记录
- 自己的销售记录、累计业绩、累计薪资
- 自己所在组的成员排名

可操作：

- 注册账号
- 登录
- 登记预发放
- 完成售出录单
- 更新自己名下识别码对应的异常状态

不可操作：

- 查看其他组详细数据
- 操作他人识别码
- 修改或撤销他人销售记录

## 7. 核心数据模型

### 7.1 `User`

在现有用户表基础上扩展：

- `role`: `ADMIN / LEADER / MEMBER`
- `remark`: 管理员备注
- `groupId`: 所属小组
- `phone`: 保留但本期不启用

规则：

- 注册默认创建 `MEMBER`
- 管理员可把成员提升为 `LEADER`
- 同一时间一个小组仅有一个组长

### 7.2 `Group`

新增小组表：

- `id`
- `name`
- `slogan`
- `leaderUserId`
- `remark`
- `status`

### 7.3 `IdentifierImportBatch`

记录识别码导入批次：

- `id`
- `sourceFileName`
- `importedByUserId`
- `successCount`
- `skippedCount`
- `createdAt`

### 7.4 `IdentifierCode`

识别码库存主表：

- `id`
- `code`
- `status`: `UNASSIGNED / ASSIGNED / SOLD`
- `importBatchId`
- `currentOwnerUserId`
- `assignedAt`
- `soldAt`

### 7.5 `CodeAssignment`

记录识别码分发历史：

- `id`
- `codeId`
- `assignedToUserId`
- `assignedByUserId`
- `assignedAt`
- `remark`

### 7.6 `ProspectDelivery`

记录“识别码已预发给某个新生”的过程数据：

- `id`
- `codeId`
- `memberUserId`
- `prospectName`
- `major`
- `qqOrWechat`
- `status`: `PRE_DELIVERED / USED / UNUSED / TRANSFERRED`
- `remark`
- `deliveredAt`
- `updatedAt`

### 7.7 `SaleRecord`

新的销售事实表以“单个识别码成交”为单位：

- `id`
- `codeId`
- `sellerUserId`
- `groupId`
- `planType`: `PLAN_40 / PLAN_60`
- `customerName`
- `major`
- `qqOrWechat`
- `saleDate`
- `remark`
- `revokedAt`
- `revokedByUserId`

一笔有效销售的事实定义：

- 绑定一个识别码
- 绑定一个套餐类型
- 未被撤销

### 7.8 `CommissionRule`

现有佣金规则模型需要升级为支持多层作用域：

- `scopeType`: `GLOBAL / ROLE / GROUP / USER`
- `role`: `LEADER / MEMBER` 可空
- `groupId` 可空
- `userId` 可空
- `tierStart`
- `tierEnd` 可空
- `price40`
- `price60`
- `effectiveStart`
- `effectiveEnd`
- `priority`

规则匹配顺序：

- 个人规则
- 小组 + 角色规则
- 角色规则
- 全局规则

## 8. 业务流程

### 8.1 小组与组长

1. 管理员创建小组
2. 成员注册账号
3. 管理员把成员分配到小组
4. 管理员从成员中指定组长
5. 组长维护小组名称和口号

### 8.2 识别码导入

1. 管理员上传 Excel/CSV
2. 前端解析识别码
3. 系统进行格式校验与去重
4. 已存在识别码跳过
5. 写入批次和识别码库存

### 8.3 识别码分发

1. 管理员选择成员
2. 管理员批量分发识别码
3. 系统更新：
   - `IdentifierCode.status = ASSIGNED`
   - `currentOwnerUserId`
   - `assignedAt`
4. 系统追加 `CodeAssignment` 历史记录

### 8.4 预发放登记

1. 成员在自己的识别码工作台里选择某个码
2. 登记预发放信息：
   - 新生姓名，可空
   - 专业，可空
   - QQ/微信，可空
   - 备注，可空
3. 系统创建 `ProspectDelivery`
4. 初始状态为 `PRE_DELIVERED`

### 8.5 成交录单

1. 成员在自己名下识别码上完成售出
2. 必须填写：
   - 套餐类型 `40 / 60`
   - 客户姓名
   - 专业
   - QQ/微信
3. 系统创建 `SaleRecord`
4. 同步更新：
   - 识别码状态为 `SOLD`
   - 预发放记录状态改为 `USED`（若存在）
   - 排行榜、汇总、薪资、组长分润实时重算

### 8.6 异常处理

- 发了但没用：
  - `ProspectDelivery.status = UNUSED`
- 发给 A，最后被 B 用：
  - `ProspectDelivery.status = TRANSFERRED`
  - 最终售出记录填写真实使用人信息

### 8.7 修改与撤销

修改：

- 只更新客户信息和备注
- 不影响业绩与薪资

撤销：

- `SaleRecord` 失效
- 识别码状态回退为 `ASSIGNED`
- 对应销量、结算、排名、组长分润全部回滚

## 9. 排行榜与统计口径

统一事实源：

- 指定时间范围内
- 所有未撤销的 `SaleRecord`

从该事实源聚合出：

- `count40`
- `count60`
- `total`
- 个人排名
- 组内排名
- 小组排名
- 管理员总榜
- 个人薪资
- 组长分润

## 10. 薪资与分润设计

### 10.1 普通成员

在指定时间范围内：

- 先统计个人累计有效卖卡数，用来命中梯度
- 再按套餐拆分计算：

`memberPay = count40 × price40 + count60 × price60`

### 10.2 组长

组长总薪资 =

1. 组长本人卖卡收入
2. 组员差额分润

组员差额分润按 `40 / 60` 分开计算：

`leaderDiff = (leaderPrice40 - memberPrice40) × memberCount40 + (leaderPrice60 - memberPrice60) × memberCount60`

### 10.3 撤销回滚

撤销后必须同步回滚：

- 成员 `count40 / count60 / total`
- 成员结算
- 组长差额分润
- 小组排名
- 全平台排名

## 11. 异常与边界规则

- 识别码导入重复：跳过
- 非法识别码格式：拒绝导入
- 已售识别码：禁止再次分发 / 再次售出
- 非持有码成员：禁止预发放、禁止录单
- 停用成员：禁止接收新识别码
- 组长只能操作本组记录
- 成员换组后：
  - 历史销售记录保留成交时的 `groupId`
  - 不随用户后续换组漂移

## 12. 迁移策略

当前旧系统以“成员直接录入 `40 / 60` 数量”为核心。

新系统采用“识别码成交记录 + 套餐类型”的事实模型。

两套模型不应强行自动互转为完全一致。

建议迁移策略：

1. 保留现有认证、后台壳层、导出、公告、横幅
2. 逐步下线旧的直接销量录入逻辑
3. 新增识别码链路后，新的销售数据全部走新模型
4. 旧数据保留只读查询能力
5. 新报表和结算主口径建立在新 `SaleRecord` 上

- 小组与组长绑定
- 识别码导入去重
- 识别码分发
- 预发放状态流转
- 成交录单
- 修改与撤销回滚
- 佣金规则匹配
- 组长差额分润计算

### 13.2 集成 / 页面

- 注册 / 登录切换
- 管理员建组、指定组长
- 管理员导入识别码、分发识别码
- 成员登记预发放并完成售出
- 组长查看本组数据并撤销记录
- 管理员查看全平台榜单、结算、导出

### 13.3 E2E 主链路

1. 成员注册
2. 管理员分配小组并指定组长
3. 管理员导入识别码
4. 管理员分发识别码
5. 成员预发放
6. 成员完成识别码售出并选择 `40 / 60`
7. 组长查看组内排名与记录
8. 管理员修改 / 撤销记录
9. 校验业绩、薪资、分润与排名同步变化

## 14. 未来预留

本期不实现，但需要为未来保留扩展方向：

- 手机号注册 + 短信验证码
- 新生 QQ / 微信 Excel 导入
- 线索批量分配给成员或小组
- 更复杂的预发放转派规则
- 更细粒度的分润报表
