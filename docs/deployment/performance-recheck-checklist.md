# 真实性能复查检查单

## 目标

- 这一轮不是泛跑 Lighthouse，而是复查当前最影响中国用户体感的真实页面。
- 优先看首屏打开速度、交互响应、登录后路由的服务器开销、榜单页的匿名访问表现。

## 当前已知基础

- `/leaderboard/daily` 已改成 public-first，不再依赖 `auth()` 和 `AppShell`。
- `/leaderboard/range` 目前是半 public-first：
  - 匿名用户可直接看榜单主内容
  - 已登录用户才展示个性化累计图表
  - 管理员才展示导出入口
- `AppShell` 的横幅和公告已经走缓存。
- 榜单、累计趋势、日节奏摘要已经走 `unstable_cache`，当前 `revalidate` 为 `30s`。

## 复查顺序

### P0：先看公开高频页

1. `/leaderboard/daily`
2. 匿名访问 `/leaderboard/range`
3. `/login`

### P1：再看成员主工作流

1. `/entry`
2. `/records`
3. 已登录访问 `/leaderboard/range`

### P2：最后看后台页

1. `/admin`
2. `/admin/members`
3. `/admin/groups`

## 每个页面都要看的核心指标

- `TTFB`
  - 先看首字节时间，优先识别服务器端慢、鉴权慢、数据库慢、缓存没命中。
- `LCP`
  - 看首屏主要内容多久稳定出现，尤其是登录页大视觉区、榜单表格首屏、后台图表区。
- `INP`
  - 看筛选、提交、切页、表单操作是否有明显卡顿。
- `CLS`
  - 看榜单、图表、顶部公告加载后是否跳动。
- `Server Duration`
  - 看具体路由函数耗时，确认慢在 SSR 还是慢在客户端。
- `Cache Hit / Miss`
  - 重点关注榜单缓存和 shell 内容缓存是否真的在吃到收益。

## 页面级检查重点

### `/leaderboard/daily`

- 目标：
  - 匿名首屏尽量接近纯公共内容页表现。
- 重点看：
  - `TTFB`
  - `LCP`
  - 日期切换后的再次打开耗时
- 异常信号：
  - 匿名访问首屏仍明显慢，说明榜单查询或缓存命中不稳定。
  - 表格区晚于标题区很久才出现，说明数据返回仍是瓶颈。

### `/leaderboard/range`

- 分开看两种访问：
  - 匿名访问
  - 已登录成员访问
- 重点看：
  - 匿名 `TTFB` 是否显著高于 `/leaderboard/daily`
  - 登录后 `TTFB` 是否因为 `auth()` 和累计图表额外拉长
  - 图表是否拖累 `LCP`
- 异常信号：
  - 匿名和登录态耗时差不多，说明匿名链路可能仍做了不必要鉴权。
  - 登录态比匿名慢很多，说明个性化累计图表查询仍是主要热点。

### `/login`

- 目标：
  - 首次进入要快，首屏大块视觉区不能拖慢可见内容。
- 重点看：
  - `LCP`
  - `INP`
  - 登录提交到跳转完成的总耗时
- 异常信号：
  - `LCP` 偏高但 `TTFB` 正常，通常是首屏视觉层和字体/样式渲染偏重。
  - 提交后卡在按钮 loading 很久，说明登录动作或跳转链路慢。

### `/entry`

- 这是当前最值得盯的登录后页面之一。
- 当前首屏会并行读取：
  - 今日记录
  - 今日节奏摘要
  - 今日目标反馈
  - 个人趋势
  - 最近提醒
- 重点看：
  - `TTFB`
  - 保存提交后的响应时间
  - `INP`
  - 重新回到页面时缓存是否生效
- 异常信号：
  - `TTFB` 明显高于榜单页，且波动大，说明多查询链路仍是首要瓶颈。
  - 提交成功后页面刷新慢，说明 revalidate 范围过大或提交后链路过重。

### `/admin`

- 重点看：
  - `TTFB`
  - 图表区域 `LCP`
  - 图表切换 preset/metric 时的交互延迟
- 异常信号：
  - 首屏慢集中在图表区，说明累计趋势查询或图表 hydration 偏重。

### `/admin/members`

- 重点看：
  - `TTFB`
  - 表格首屏渲染
  - 新增/编辑成员后的返回速度
- 异常信号：
  - 大表格页打开慢但数据量不大，通常是 SSR 查询和客户端表格 hydration 叠加。

### `/admin/groups`

- 重点看：
  - `TTFB`
  - 编辑小组或组长后的响应和回显速度
- 异常信号：
  - 提交后页面长时间等待，说明动作完成后的重查和重渲染成本仍偏高。

## 复查维度

- 设备：
  - 手机优先
  - 桌面端补充
- 身份：
  - 匿名
  - 成员
  - 管理员
- 场景：
  - 首次打开
  - 二次打开
  - 提交后回流

## 建议阈值

- `TTFB`
  - P0 页面优先压到 `500ms` 内，最差不超过 `800ms`
  - 登录后页面优先压到 `800ms` 内，最差不超过 `1200ms`
- `LCP`
  - `<= 2.5s`
- `INP`
  - `<= 200ms`
- `CLS`
  - `<= 0.1`

## 这轮最值得优先盯的差异对比

1. `/leaderboard/daily` 匿名 vs `/leaderboard/range` 匿名
2. `/leaderboard/range` 匿名 vs `/leaderboard/range` 登录态
3. `/login` 首屏 vs 登录提交后跳转
4. `/entry` 首次打开 vs 保存后回流
5. `/admin` 首页 vs `/admin/members` 大表格页

## 如果复查后要继续做下一刀优化

- 第一优先：继续压 `/leaderboard/range` 匿名链路，尽量避免匿名流量走完整鉴权成本。
- 第二优先：压 `/entry` 首屏多查询开销，确认哪些数据必须首屏拿，哪些可以延后。
- 第三优先：压 `/admin` 图表和大表格页的首屏体感。
